Frogger-Inspired Arcade Game

My very first game inspiration

Try my game here:
https://good-herobrine-776.github.io/frogger/

 

Info

This version of Frogger is a recreation of the original 1981 game.

In this game, you avoid the cars, use the platforms to cross the river, and reach the 5 goals. If you lose 3 lives, the game is over.

This game is made to mostly match the original game's first level, with some tweaks added to this spinoff.

The Process


As I programmed this game, I learned how to connect different scripts together, including the UI, Game Manager, and PlayerController. From the editor, I was also able to call components from specific GameObjects for use. I also organized my code by creating different scripts to manage different game functionality. This helps me easily keep track of which scripts are carrying out specific functions.

Using the editor was quite an experience as I explored the use of a sprite sheet, sound controller, animator, renderers and colliders. More importantly, I learned how to reference them in the scripts such that these editor components change to reflect specific events in the game, such as winning, losing a life, or movement.

Learning all of this has made me more confident in making more games, and also gave me a further idea on how I can program games while staying organized.


Game Features

Objective

The goal is the same as the original game, you reach the 5 goals without losing 3 frogs.



Additions &
Modifications

Most of the features are to match the original, but with a few tweaks. There is background music, some spawn rates are altered, and disappearing turtles are uniquely-colored.


The same features

Features like the UI are kept the same, as the score increments are the same as the original, sounds, the cars spawn rates, and most of the colors and sprites.

Reflection


Upon finishing the game, it felt very accomplishing as I acquired more expertise in C# coding and Unity2D. What I'm also proud of is how I kept everything organized by making specialized scripts and splitting my bodies of code through methods that are referenceable by other scripts, such that it is easy to follow, edit and debug. However, the hardest parts were the colliders and the logic as the collider events were unpredictable; while testing, sometimes colliders would be too big, trigger randomly or not work as expected. From those setbacks, I learned how to persist through failure by spending more time to plan, make necessary edits and use the debugger carefully. Overall, it boosted my confidence and it was a satisfactory introduction to game development.
